### Research: React 19 `useOptimistic` for Comment Threads [ux] [react]

**Key insight:** `useOptimistic` (React 19) enables instant UI updates that assume success, rolling back on failure. For comment threads this means: comment appears immediately with a "Sending..." indicator, confirmed on success, removed + error toast on failure.

Pattern used in the ticket detail page:
```tsx
const [optimisticComments, addOptimisticComment] = useOptimistic(
  comments,
  (state, newComment) => [...state, newComment]
)
// On submit: addOptimisticComment(optimisticComment) → instant
// On server response: dispatch({ type: "replace", tempId, real }) → confirm
```
**Why it matters for this portal:** Support tickets are high-anxiety for clients. Immediate comment confirmation removes the "did my message go through?" question. The portal feels native and responsive.

## 4:00 PM Session — Client Portal: Phase 2 Scaffold + Stripe Webhook

### Research

- Studied how embedded Sanity Studio works in Next.js App Router: `NextStudio` is a Client Component wrapping Sanity's `StudioProvider`/`StudioLayout`. The config is a plain serializable object — not a class instance — so it can be built in a Server Component and passed as a prop.
- Key architectural finding: for multi-tenant, use `force-dynamic` (not the docs' `force-static`) because the Sanity project ID varies per tenant. The config factory pattern (Server Component) → `NextStudio` (Client Component) is the right architecture.
- Studied the `moduleResolution: "bundler"` + pnpm nested node_modules problem: TypeScript can't resolve `sanity` at compile time even though Node.js resolves it correctly at runtime. Solution: `// @ts-ignore` on dynamic import.
- Studied Stripe webhook design: always return 200 to Stripe even on CRM failure (no retry storms). Portal is a relay; CRM is authoritative.

## 6:00 PM Session — Phase 4 Stripe Portal Fix + Phase 2b Studio Plugins

### Research

**Multi-tenant SaaS DB patterns [arch]:**
- Three patterns: shared DB + `tenant_id` (current portal approach), separate schemas per tenant (migration complexity), separate DBs (max isolation, max ops overhead)
- PostgreSQL Row-Level Security (RLS) is the pragmatic upgrade: DB-level `SET app.current_tenant = 'acme'` + RLS policies check it. Even if app code has a bug, DB prevents cross-tenant reads.
- Current portal uses shared DB + app-level `tenantId` filtering (correct for scale). Future hardening: add RLS as second line of defense.
- Key Stripe finding: Customer Portal flow requires server-side `stripe.billingPortal.sessions.create()` + `redirect(url)`. NOT a client-side redirect to a pre-stored URL.

**Sanity Studio structureTool + visionTool [ux]:**
- `structureTool` uses Structure Builder API (`@sanity/structure`) — `S.list()` with `S.documentTypeListItems()` auto-generates a list for every registered schema type (key for multi-tenant: we don't know which types each tenant has)
- `visionTool` (`@sanity/vision`) embeds GROQ playground inside the Studio
- `defineConfig` returns a plain object — `plugins: [structureTool({ structure }), visionTool()]` is all that's needed
