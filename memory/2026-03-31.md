### Research: React 19 `useOptimistic` for Comment Threads [ux] [react]

**Key insight:** `useOptimistic` (React 19) enables instant UI updates that assume success, rolling back on failure. For comment threads this means: comment appears immediately with a "Sending..." indicator, confirmed on success, removed + error toast on failure.

Pattern used in the ticket detail page:
```tsx
const [optimisticComments, addOptimisticComment] = useOptimistic(
  comments,
  (state, newComment) => [...state, newComment]
)
// On submit: addOptimisticComment(optimisticComment) ‚Üí instant
// On server response: dispatch({ type: "replace", tempId, real }) ‚Üí confirm
```
**Why it matters for this portal:** Support tickets are high-anxiety for clients. Immediate comment confirmation removes the "did my message go through?" question. The portal feels native and responsive.

## 4:00 PM Session ‚Äî Client Portal: Phase 2 Scaffold + Stripe Webhook

### Research

- Studied how embedded Sanity Studio works in Next.js App Router: `NextStudio` is a Client Component wrapping Sanity's `StudioProvider`/`StudioLayout`. The config is a plain serializable object ‚Äî not a class instance ‚Äî so it can be built in a Server Component and passed as a prop.
- Key architectural finding: for multi-tenant, use `force-dynamic` (not the docs' `force-static`) because the Sanity project ID varies per tenant. The config factory pattern (Server Component) ‚Üí `NextStudio` (Client Component) is the right architecture.
- Studied the `moduleResolution: "bundler"` + pnpm nested node_modules problem: TypeScript can't resolve `sanity` at compile time even though Node.js resolves it correctly at runtime. Solution: `// @ts-ignore` on dynamic import.
- Studied Stripe webhook design: always return 200 to Stripe even on CRM failure (no retry storms). Portal is a relay; CRM is authoritative.
